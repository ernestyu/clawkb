# 从收藏到洞察：clawsqlite + 兴趣簇，让知识库自己长出结构

> 我不缺收藏夹，我缺的是「我到底在关心什么」的那张地图。


## 0. 引子：收藏夹地狱

如果你也习惯「看见好文章就先存一下」，大概会有这些熟悉的画面：

- 浏览器收藏夹一层套一层，真正点开过的屈指可数；
- 各种稍后读、微信收藏、Notion/Obsidian 里，塞满了「以后要好好读」的东西；
- 偶尔想起来搜一搜，却发现只能记得几个模糊关键词，结果不是搜不到，就是翻不完。

这些工具本身并不差：

- 它们解决了「**存下来**」的问题；
- 也部分解决了「**按关键字找到**」的问题；
- 但几乎没有哪一个在认真回答：
  - 我到底在关心什么？
  - 过去这段时间，我在什么方向上看得多、在哪些方向上几乎没有更新？
  - 我的知识库，能不能自己长出一张「兴趣地图」，而不是一堆标签和文件夹？

这篇文章讲的是，我在 `clawsqlite` 这个本地知识库上做的一些实践：

1. 用向量检索补齐关键词搜索的盲区；
2. 用 1024 维 embedding 空间做两阶段兴趣聚类，让知识库自己长出「兴趣簇」；
3. 基于兴趣簇做一条每周的「知识库兴趣周报」，让你知道：
   - 我最近到底收了多少东西？
   - 这些东西落在什么兴趣簇？
   - 哪些簇在升温，哪些簇在慢慢降温？

核心工具是 `clawsqlite`，一个 Markdown + SQLite 的轻量知识库 CLI。后面所有例子，都围绕它来展开。


## 1. 基础设施：clawsqlite 这个「知识仓库」

先简单说一下 `clawsqlite` 本身，不熟悉的同学可以当成背景介绍，熟悉的可以直接跳到向量和聚类部分。

### 1.1 什么是 clawsqlite

`clawsqlite` 是一个围绕 SQLite 打造的 CLI 工具箱，目前主要关注一个应用：本地 Markdown + SQLite 知识库。

它的核心想法很简单：

- **正文**：每篇文章一份 Markdown 文件，放在 `articles/` 目录下；
- **结构化记录**：在 SQLite 的 `articles` 表里存 title / summary / tags / category / 路径 / 时间戳等等；
- **全文索引**：把标题、标签、摘要等写入 FTS5 索引，做关键词搜索；
- **向量索引（可选）**：对摘要和标签做 embedding，写入 `articles_vec` / `articles_tag_vec`，做向量检索。

换句话说，`clawsqlite` 做的是：

> 把「我散落在各处的文章」统一收拢，变成「文件系统 + SQLite + FTS +（可选）vec0」这一套基础设施。

### 1.2 一条最小的工作流

你可以把 clawsqlite 视为一个 CLI 应用：

```bash
clawsqlite knowledge --help
```

最小闭环大概像这样：

```bash
# 1. 写一条纯文本笔记 / 文章
clawsqlite knowledge ingest \
  --text "你好，clawsqlite" \
  --title "第一次笔记" \
  --category test \
  --tags demo \
  --gen-provider off \
  --json

# 2. 在知识库里搜索
clawsqlite knowledge search "clawsqlite" --mode fts --json
```

这两条命令会帮你：

- 创建 `<root>/knowledge.sqlite3`；
- 在 `<root>/articles/` 下生成一份 Markdown 文件；
- 建好 FTS 索引；
- 如果你配置了 Embedding 服务，还会顺手把摘要写入 `articles_vec` 表。

知识库有了，全文搜索也有了，embedding 也有了——看起来已经可以用很久。但在引入兴趣簇之前，我还是先讲讲：为什么单靠关键词搜索是不够的。


## 2. 关键字 vs 向量：先解决「怎样找到」的问题

### 2.1 关键字搜索的边界

传统笔记/收藏系统，大部分只提供关键词搜索：

- 你得记得那几个词，才能把相关内容翻出来；
- 如果你模糊记得的是「一个关于个人读报雷达的架构讨论」，而不是具体标题，那就很难；
- 更别提一个抽象的兴趣，比如「大模型工具链里那些和数据流相关的文章」，很难用几个 keyword 说清楚。

关键字搜索最大的问题，是它假定：

> 人能用近似的字面表达，来精确锁定一组内容。

但我们真正的查询往往是模糊的、半记忆的、跨领域的，这就给向量检索留下了空间。

### 2.2 clawsqlite 里的混合检索

在 clawsqlite 里，我做了一套 hybrid search，把 FTS 和向量组合了起来：

1. **长摘要 embedding**：
   - 每篇文章会生成一个长摘要（约 1200 字左右），作为语义的主载体；
   - 这个摘要被 embed 到 `articles_vec` 里。
2. **标签 embedding**：
   - tags 本身也会被 embed 到 `articles_tag_vec`；
   - 标签通道在打分里有独立权重，不完全跟摘要捆在一起。
3. **查询侧 embedding**：
   - 自然语言 query 先过一遍轻量关键词抽取；
   - 然后把「原始 query + 关键词 tail」一起 embed，减少虚词干扰，同时保留上下文；
4. **打分**：
   - 用一个 logistic sigmoid 归一化向量距离，避免「还行」的相似度把「真正近邻」压扁；
   - 最终打分大致是：

     ```text
     score = vec + fts + tag_sem + tag_lex + priority + recency
     ```

效果是：

- 搜索「个人读报雷达」，不仅能命中直接提到这几个字的文章；
- 还会把「有关 OpenClaw pipeline 设计」「embedding timeout 降级」「日志设计」这些相关内容一起拉出来。

这一层解决的是「**如何找到内容**」的问题。

但我很快发现，这还不够。

> 就算我能找到任何一篇文章，我依然不知道：**整个知识库的形状是什么**。


## 3. 新问题：我在关心什么？

有了一个稳定的知识库之后，我脑子里的问题变成了：

- 我收集了这么多东西，它们大致可以分成哪些「兴趣方向」？
- 哪些方向是长期稳定关注的？
- 哪些方向是最近突然冒出来的？
- 哪些方向是我曾经很感兴趣，现在几乎不更新的？

这些问题很难靠「标签」和「分类」回答：

- 起标签名本身就是一件麻烦事；
- 很多人一开始会用 `ai / ml / llm`，后来又加 `openclaw / tooling / pipeline`，久而久之完全混在一起；
- 更糟的是，即便你很勤奋地打 tag，也很难一眼看出「这些标签之间的空间结构」。

我更想要的是一种**自动长出来的兴趣结构**：

- 不需要我手工给每篇文章贴「这是 prompt engineering」「那是 embedding 架构」；
- 只要 embedding 的空间是有结构的，就应该能在 1024 维里长出一些自然的「兴趣簇」。

这就是后面这套「兴趣簇（interest clusters）」设计的出发点。


## 4. 兴趣簇：在 1024 维空间里画出兴趣地图

这一节稍微技术一点，但我会尽量用直白的语言来描述。

### 4.1 聚类的素材：我们拿什么做簇？

在 clawsqlite 的知识库里，每篇文章至少有一条重要的向量：

- 由摘要（summary）生成的 1024 维 embedding；
- 有时会再加上一份 tags 的 embedding，但聚类的主空间仍然是 summary embedding。

我们在设计时特意限制了一些「看起来很酷但不稳定」的做法：

- 不在默认流水线中做 PCA 截断（不会先把 1024 降到 64 再聚类）；
- 不做 t-SNE/UMAP 之类只适合可视化的非线性降维；
- 聚类在**原始 1024 维空间**里完成，避免冷门小簇被主成分「舔平」。

降维只用在一个地方：

- 为了在报告里画一张 2D 的 PCA 散点图，帮助人类理解簇心的大致分布。

### 4.2 两阶段聚类算法：先打散，再合并

真正的兴趣簇构建，大致分两阶段：

#### 阶段一：高 k 的 k-means + 小簇重分配

- 用一个相对偏大的 k（受 `max_clusters` 和数据量约束）在 1024D 空间跑一轮 k-means；
- 对非常小的簇（少量文章）做「小簇重分配」：
  - 如果一个簇成员数低于 `min_size`，就把这些点挪到最近的大簇里；
- 得到一批「还算紧凑」的候选簇心。

直观上，这是在砸碎「大锅粥」：

- 把显然差异很大的主题先硬拆开；
- 再把明显过于鸡肋的小簇合入更可信的大簇。

#### 阶段二：基于簇心余弦距离的 union-find 合并

有了候选簇心以后，我们再用一套更温和的逻辑来「合并相近兴趣」：

- 定义簇心距离：

  ```text
  d(center_i, center_j) = 1 - cos_sim(center_i, center_j)
  ```

- 设定一个 merge 阈值 `T = CLAWSQLITE_INTEREST_MERGE_DISTANCE`：
  - 当 `d < T` 时，认为这两个簇语义上可以合并；
  - 所有满足条件的簇用 union-find 合并成 group；
- 每个 group 重新计算一个新的中心，作为最终的兴趣簇心。

这样做的好处是：

- 第一阶段负责「把东西砸碎」；
- 第二阶段负责「在空间上重新粘合」，把本来就应该在一起的 topic 合回去。

### 4.3 参数与调参：如何控制簇的颗粒度

我们给兴趣簇相关的参数设计了统一优先级：

- CLI 参数 > `.env` 中的 `CLAWSQLITE_*` > 代码默认值。

主要参数包括：

- `min_size` / `CLAWSQLITE_INTEREST_MIN_SIZE`（默认 5）
  - 每个簇的最小文章数，小于该值的小簇会被合并到最近的大簇；
- `max_clusters` / `CLAWSQLITE_INTEREST_MAX_CLUSTERS`（默认 16）
  - 第一阶段 k-means 的簇数量上限；
- `CLAWSQLITE_INTEREST_MERGE_DISTANCE`（默认内部 fallback 0.06）
  - 簇心之间 `1 - cos` 的合并阈值；
  - 越小越「细腻」，越大越「粗」，最终簇数也越少；
- `CLAWSQLITE_INTEREST_MERGE_ALPHA`（默认 0.4）
  - 用真实簇结构自动推一个建议阈值：

    ```text
    merge_suggest ≈ alpha * median(mean_radius)
    ```

    中位数 mean_radius 代表典型簇内半径，alpha 控制「多粘一点」还是「更保守」。

在真实知识库上跑下来，我们会看：

- 每簇 `mean_radius` / `max_radius` 的分布；
- 簇心两两 `1 - cos` 的 `min / median / max`；
- PCA 2D 图里，点的颜色代表 `mean_radius`，大小代表簇大小；

通过这些指标，可以判断这套参数下，是不是已经变成「另一锅粥」，还是能看到一组有辨识度的兴趣簇。


## 5. 从簇到报告：每周给知识库做一次「兴趣体检」

有了兴趣簇，还只是「静态结构」。我更想要的是一份**动态报告**：

- 告诉我最近一段时间，知识库发生了什么；
- 告诉我在哪些兴趣簇上投入最多；
- 告诉我哪些方向在变热，哪些方向已经很久没更新。

这就是 `report-interest` 子命令要做的事情。

### 5.1 报告想回答的问题

一个最简单的周报应该至少回答这些：

- 这周我一共收了多少篇文章？
- 每天的新增分布是什么样的？是否有几天完全空窗？
- 这些文章分别落在哪些兴趣簇？
  - 每个簇里大概有多少篇？占比多少？
  - 每个簇内部是不是很散？是不是在变得越来越「大锅粥」？
- 相比上一周，哪些簇明显升温、哪些簇明显降温？

有了这些信息，你才能对自己的知识库有一点「时间维度」的感觉。

### 5.2 `report-interest` 的 CLI 设计

我们给 clawsqlite 的知识库 CLI 增加了一个子命令：

```bash
clawsqlite knowledge report-interest \
  --db /home/node/.openclaw/workspace/orgmode/clawsqlite/data/clawkb.sqlite3 \
  --days 7 \
  --vec-dim 1024 \
  --lang zh \
  --format html \
  --out-dir /home/node/.openclaw/workspace/orgmode/clawsqlite/reports
```

参数含义：

- `--db`：知识库 SQLite 路径；
- `--days`：回溯的时间窗口（这里是过去 7 天）；
- `--vec-dim`：embedding 维度（大多数 bge/m3 类模型是 1024）；
- `--lang`：报告框架语言（目前支持 `en` / `zh`，Markdown 中的标题、小节名、表头会跟着语言切换）；
- `--out-dir`：报告根目录（默认是当前工作目录下的 `./reports`）；
- `--format`：额外输出格式：
  - 缺省 / `md`：只生成 Markdown + PNG 图；
  - `html`：在此基础上再生成一个自包含的 HTML 报告；
- `--no-pdf`：不尝试生成 PDF（默认会用 pandoc 试一次，失败不报错）。

命令跑完后，会在 `--out-dir` 下生成一个以窗口结束日期命名的目录，比如：

```text
/home/node/.openclaw/workspace/orgmode/clawsqlite/reports/
  20260331-report/
    report.md
    report.html      # 若 --format html 且 pandoc 存在
    report.pdf       # 若环境有 LaTeX 且未加 --no-pdf
    images/
      daily_articles.png
      cluster_distribution.png
      interest_clusters_pca.png
```

Markdown + PNG 是**始终保证**的产物；PDF / HTML 都是「尽力而为」，缺少依赖时不会把命令搞挂。

### 5.3 报告里都写了什么？

下面按模块大致介绍一下报告内容：

#### 5.3.1 总览：时间窗口 & 基本指标

报告开头是一段很短的概览：

- 时间窗口：比如 `2026-03-25 - 2026-03-31`；
- 新增文章数：比如 `16`；
- 涉及兴趣簇数：比如 `7`（这周至少有 7 个簇新增了内容）。

这部分是「一眼看完」的指标，方便你快速知道这周有没有在看东西，以及是不是只集中在极少数簇上。

#### 5.3.2 每日新增文章数

第二部分是一个简单的表格 + 一张柱状图：

- 表格：每一行是 `日期 / 新增文章数`；
- 图表：横轴为日期，纵轴为新增数。

这个图很实用：

- 如果你打算保持「每天至少读一点」的节奏，一眼就能看出哪些天是纯空白；
- 如果某几天突然爆量（比如某个 topic 突然上头），也会非常明显。

#### 5.3.3 按兴趣簇分布

这一节是整个报告的核心部分之一。

先是一张汇总表：

- 簇 ID；
- 主题概括（由 tags + keywords 选出来最多 5 个 token，用 `/` 拼接）；
- 文章数；
- 占比（相对于整个窗口的总新增）；
- `mean_radius`（簇内平均 `1 - cos` 距离，越小越紧凑）。

接着是一张「按兴趣簇分布」的柱状图：

- 横轴是簇 ID；
- 纵轴是该簇在本窗口内的文章数。

然后，每个簇会单独展开一个小节，列出最近的代表文章，例如：

```text
#### Cluster 7 - TurboQuant/GoogleResearch/PolarQuant/KV cache/H100加速
- [2026-03-27] H100 上 KV cache 加速的一些实践
- [2026-03-29] PolarQuant 论文笔记
...
```

这里有几个设计要点：

- 主题概括并不是从全文中瞎取词，而是优先用 tags，再用轻量关键词补齐，控制在 5 个以内；
- 日期 + 标题的形式，方便你回到 `clawsqlite knowledge show` 或原始 Markdown 里去看细节；
- 对于只有 1–2 篇文章的小簇，也会给出代表列表，帮助你发现一些「刚冒头的新兴趣」。

#### 5.3.4 簇心 PCA 散点图

这一张图更偏「全局俯视视角」：

- 我们把所有兴趣簇心做一次 PCA 降到 2D，仅用于绘图；
- 每个点代表一个簇心：
  - 坐标是 (PC1, PC2)；
  - 点大小 ∝ sqrt(簇大小)；
  - 颜色映射 `mean_radius (1 - cos)`。

直观上：

- 某些点群抱在一起，说明这些簇语义很接近，也许是一个更大的主题的子簇；
- 某些点远离主云团，可能是一个非常独立的小众兴趣（比如某种特别的硬件/某个具体项目）。

#### 5.3.5 升温 / 降温簇

最后一部分，是「趋势」：

- 把当前时间窗口和前一个等长窗口做对比，比如「本周 vs 上周」；
- 对每个簇统计两个窗口内的文章数，并计算差值 `Δ`；
- 用一张小表格展现：

```text
| 簇 ID | 主题概括 | 当前窗口新增 | 上一窗口新增 | 变化 |
|  1    | ...      |      5      |      1      | +4  |
|  7    | ...      |      1      |      4      | -3  |
```

这样，你一眼就能看到：

- 哪几个簇在最近一周明显升温（正向 Δ 较大）；
- 哪几个簇在降温（负向 Δ 较大）；
- 有没有某个曾经很热的方向，已经连续几周是 0 了。


## 6. 从收藏到洞察：这套东西到底改变了什么？

回到一开始的问题：我们并不缺「收藏工具」，我们缺的是**对于自己知识结构的洞察**。

这套 `clawsqlite + 兴趣簇 + 周报` 做到了几件事：

1. **从“能搜到”到“看得见结构”**
   - 向量搜索解决了模糊概念和语义相近内容的查找问题；
   - 兴趣簇让你看到：这些内容在 1024 维空间里长成了哪些「岛屿」。

2. **从“有堆积”到“有节奏”**
   - 每周兴趣报告，至少会迫使你每周看一眼：
     - 这周收了多少？
     - 是不是某几天完全空窗？
     - 是否一直在同一个 topic 打转？

3. **从“我在记什么”到“我在关心什么”**
   - 传统笔记/收藏更像是「记帐」：记下今天看到的东西；
   - 兴趣簇和趋势报告，开始回答「我长期关注的是哪些方向」；
   - 这对于做长期规划（比如职业方向、研究主题、投资方向）其实是很有价值的侧写。

4. **对后续自动化的支点**
   - 有了稳定的兴趣簇，你可以在其它工具里复用这套结构：
     - 读报雷达可以按兴趣簇做更合理的多样性选取，而不是简单 top-K；
     - 推荐系统可以基于兴趣簇做冷启动，而不是只看最近几条点击；
     - 定时提醒可以变成「你已经 3 周没有新增 `cluster 3: OpenClaw/架构/工具链` 相关内容了」。


## 7. 小结 & 展望

这次在 clawsqlite 上做的改动，大概可以用一句话概括：

> 从「能存 / 能搜」走向「能理解自己在存什么」。

它不是一套华丽的可视化，而是一条相对克制、纯 CLI + Markdown + PNG 的路径：

- 兴趣簇的算法尽量简单可解释，不依赖复杂的深度模型和黑盒降维；
- 所有参数都有清晰的优先级与默认值，可以通过 CLI 和 `.env` 来调整；
- 报告输出是你能用任何工具二次处理的 Markdown + 图表，而不是一个专有格式。

接下来，还有很多可以迭代的方向，例如：

- 按月/季度生成更长周期的兴趣迁移图；
- 把周报通过 bot 自动推送到 Telegram/Signal；
- 在 clawfeedradar 里直接挂一个「按兴趣簇阅读」模式；
- 在报告里加上更多「质量」维度的指标，比如每簇的平均阅读时长、标星率等（如果有这些数据）。

但对我个人来说，现在这一步已经足够有用：

- 它让我每周至少认真看一次自己的知识库；
- 它让「收藏」这件事从单纯的累积，变成了一种可以被观察、被调整的行为。

如果你也有一堆散落在各处的笔记和收藏，不妨给它们一个机会，让它们在一个干净的 SQLite 里，自己长出一点结构来。哪怕只是一张每周更新的兴趣分布图，长期看下去，也会比「收藏夹地狱」带来更多的东西。  

（完）
